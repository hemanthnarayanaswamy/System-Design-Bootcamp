# "What Happens When You Type a URL Into a Browser?" — FAANG Prep Guide

## Part 1: The Script

This is structured so you can deliver it in ~5 minutes for the high-level pass, then go deep where the interviewer pulls you.

---

### Opening (the framing move)

> "There's a lot of layers to this — URL parsing, DNS, transport, TLS, HTTP, server-side, then browser rendering. Let me do a high-level pass first and then we can go deep wherever you're most interested. Quick clarifying question — are we assuming HTTPS, a cold cache, and a modern browser like Chrome? And should I treat the server as a single origin, or is it fine to assume there's a CDN in front?"

Why this works: you're showing scope management, signaling there's nuance, and inviting the interviewer to steer. It also buys you 10 seconds to organize your thoughts.

Assume they say "yes, HTTPS, cold cache, CDN is fine." Now go.

---

### Stage 1 — URL parsing and pre-flight checks

> "First, the browser parses the URL into its components — scheme, host, optional port, path, query string, fragment. A few things happen before any network call:
>
> - The omnibox decides whether this is actually a URL or a search query.
> - If the host is on the HSTS preload list, the browser rewrites `http://` to `https://` before it ever hits the network — this prevents downgrade attacks.
> - International domain names get punycode-encoded.
> - The browser checks if it already has a service worker registered for this origin that might intercept the request."

### Stage 2 — Cache checks

> "Before going to the network, the browser checks a stack of caches in order: the in-memory cache, then the service worker if one is registered, then the HTTP disk cache. If there's a fresh response, we short-circuit and return it. If we have a stale response with an ETag or Last-Modified, we'll do a conditional request later with `If-None-Match` and potentially get a 304."

### Stage 3 — DNS resolution

> "Assuming we need the network, we resolve the hostname to an IP. There's another cache hierarchy here:
>
> - Browser DNS cache (Chrome holds entries ~60 seconds).
> - OS resolver cache, plus the local hosts file.
> - Router cache.
> - The ISP's recursive resolver — or, increasingly, a DoH resolver like 1.1.1.1 or 8.8.8.8 over HTTPS, which keeps the query encrypted from the ISP.
>
> If nothing is cached, the recursive resolver walks the hierarchy: root nameservers, then the TLD nameserver (`.com`), then the authoritative nameserver for the domain, which returns the A or AAAA record. In practice, that A record probably points at a CDN edge, not the origin server."

### Stage 4 — TCP handshake (or QUIC)

> "Now we open a connection. For HTTP/1.1 or HTTP/2, that's a TCP three-way handshake — SYN, SYN-ACK, ACK — costing one round trip before we can send anything. TCP also starts in slow-start, so initial throughput is limited.
>
> If the server supports HTTP/3, we go over QUIC, which runs on UDP and combines the transport handshake with the crypto handshake — saving a round trip and avoiding TCP head-of-line blocking, which I can come back to."

### Stage 5 — TLS handshake

> "Over that connection, we do a TLS handshake. With TLS 1.3, this is one round trip: the client sends a ClientHello with its supported cipher suites, key share, and the SNI indicating which hostname it wants — SNI matters because one IP often hosts many sites. The server responds with its certificate, its key share, and a finished message. The client validates the cert chain against its trust store — leaf cert signed by intermediate, intermediate signed by a trusted root CA — checks the hostname, expiration, and revocation via OCSP stapling.
>
> If we've connected before, TLS 1.3 supports 0-RTT resumption using a pre-shared key, so we can send application data in the very first packet — though that has replay-attack tradeoffs."

### Stage 6 — HTTP request

> "Now we send the actual HTTP request: method, path, headers — including the Cookie header, User-Agent, Accept, Accept-Encoding telling the server we support gzip or brotli.
>
> If we're on HTTP/2, headers are compressed with HPACK and the request is one stream multiplexed over the connection — meaning multiple requests can be in flight on a single TCP connection without head-of-line blocking at the application layer. HTTP/3 uses QPACK and avoids head-of-line blocking at the transport layer too, since QUIC streams are independent."

### Stage 7 — Server-side processing

> "I'll abstract this since it's outside the browser, but typically: the request hits a CDN edge first. If the resource is cacheable and fresh at the edge, we get a response without ever touching the origin. Otherwise the edge forwards to the origin — usually through a load balancer, then a reverse proxy like nginx, to an application server, which may hit a cache like Redis and a database before composing a response."

### Stage 8 — HTTP response

> "The server returns a status code, headers, and a body. Headers worth calling out:
>
> - `Cache-Control` and `ETag` — control downstream caching.
> - `Content-Encoding` — likely brotli for HTML these days.
> - `Strict-Transport-Security` — pins HTTPS for future visits.
> - `Content-Security-Policy` — restricts what the page can load and execute.
> - `Set-Cookie` — for session state."

### Stage 9 — Browser rendering

> "Now the browser turns bytes into pixels. The pipeline:
>
> 1. **Parse HTML into the DOM tree.** Streaming — the browser starts parsing as bytes arrive.
> 2. **Parse CSS into the CSSOM.** CSS is render-blocking — the browser won't paint until it has the CSSOM.
> 3. **Execute JavaScript.** Inline or non-async/non-defer scripts block the HTML parser. `async` scripts execute as soon as they download. `defer` scripts wait until parsing is done. JS can mutate the DOM and CSSOM, which is why it's parser-blocking by default.
> 4. **Build the render tree** — DOM + CSSOM, minus invisible nodes like `<head>` or `display: none`.
> 5. **Layout (reflow).** Compute the geometry of every node — positions, sizes — based on viewport.
> 6. **Paint.** Fill pixels into layers — text, colors, borders, images.
> 7. **Composite.** The GPU combines layers into the final frame. Transforms and opacity are cheap because they happen at the compositor without re-running layout or paint.
>
> Meanwhile, every subresource the parser encounters — CSS, JS, images, fonts — kicks off its own request cycle, possibly reusing the existing connection."

### Closing the high-level pass

> "And then there's a long tail — `DOMContentLoaded` fires when the DOM is parsed, `load` fires when subresources finish, and Core Web Vitals like LCP measure when the largest content element paints. Happy to go deeper on any layer — DNS, TLS, the rendering pipeline, or the HTTP version differences are usually where I have the most to say."

That last sentence is a tee-up — it tells the interviewer where you're strongest and lets them pick.

## Part 2: The Follow-Ups

These are the questions you'll almost certainly get. Have a strong answer for each.

---

### Follow-up 1 — "The page is slow. How would you debug it?"

Walk through it as a methodology, not a checklist:

> "I'd start by figuring out _which_ slow — first paint, interactivity, or steady-state. The Core Web Vitals frame this well: LCP for loading, INP for interactivity, CLS for visual stability.
>
> Then I'd open DevTools and look at the network waterfall. The questions I'm asking:
>
> - **Is TTFB high?** That's a server or CDN issue — maybe a cache miss, slow DB query, or distant origin.
> - **Is the connection setup expensive?** Look at DNS, TCP, TLS phases of the first request. Fix with preconnect, HTTP/3, or a closer edge.
> - **Are render-blocking resources delaying first paint?** Critical CSS inline, defer non-critical JS, async third-party scripts.
> - **Is the LCP image lazy-loaded or large?** Preload it, serve responsive sizes, use modern formats like AVIF or WebP.
> - **Is JavaScript blocking the main thread?** Performance tab flame chart shows long tasks. Code-split, defer, move work to web workers.
> - **Layout thrashing?** Look for forced synchronous layouts in the performance tab — usually JS reading layout properties in a loop after writes.
>
> I'd use Lighthouse for a starting checklist but trust the actual waterfall and flame chart over the score."

This answer signals real production experience.

---

### Follow-up 2 — "What does a TLS certificate actually contain, and how is it verified?"

> "A cert is an X.509 document. The important fields are:
>
> - **Subject** — who the cert is for, including the common name and Subject Alternative Names (SANs). Modern browsers ignore CN and use SANs.
> - **Issuer** — which CA signed it.
> - **Validity period** — not-before and not-after dates.
> - **Public key** — the server's public key.
> - **Signature** — the issuer's signature over the cert's contents.
>
> Verification: the server sends its leaf cert plus intermediates. The browser walks the chain — leaf signed by intermediate, intermediate signed by root — until it hits a root CA in its trust store. Each signature is verified using the parent's public key.
>
> Then it checks: hostname matches a SAN, the chain is within validity, no cert in the chain is revoked (via OCSP, ideally stapled by the server to avoid a side-channel request), and the cert is logged in Certificate Transparency logs.
>
> What the cert proves: that whoever holds the private key controls the hostname at the time the CA issued the cert. It does _not_ prove the server isn't compromised right now — it just proves identity."

---

### Follow-up 3 — "What's different on a second visit?"

> "Almost every layer caches:
>
> - **DNS** — cached at the OS and browser, so no resolver lookup.
> - **TCP** — if the connection is still open (keep-alive), we skip the handshake. Otherwise we still pay it.
> - **TLS** — session resumption with a pre-shared key, often 0-RTT in TLS 1.3, so we send the request in the first packet.
> - **HSTS** — first visit might have been HTTP-redirected to HTTPS, costing a round trip. Now the browser goes straight to HTTPS.
> - **HTTP cache** — fresh resources return from disk. Stale-but-valid resources do a conditional GET and likely get a 304 Not Modified, saving the body transfer.
> - **Service worker** — can serve from its own cache offline-first, with no network at all.
> - **Compiled JS** — V8 caches bytecode so JS parses and compiles faster.
>
> The net effect: a well-cached return visit can be an order of magnitude faster than the cold visit, and that's why caching is the single biggest performance lever on the web."

---

### Follow-up 4 — "What's the difference between HTTP/1.1, HTTP/2, and HTTP/3?"

> "HTTP/1.1 — text protocol, one request per connection at a time. Browsers work around this by opening 6 parallel TCP connections per origin. Suffers from head-of-line blocking — a slow response blocks everything behind it on that connection.
>
> HTTP/2 — binary protocol, multiplexed streams over a single TCP connection. HPACK header compression. Removes application-layer head-of-line blocking, but TCP-level head-of-line blocking remains: a single dropped packet stalls all streams because TCP delivers bytes in order.
>
> HTTP/3 — runs over QUIC, which is on UDP. Streams are independent at the transport level, so a dropped packet in one stream doesn't block the others. Combined transport + TLS 1.3 handshake reduces connection setup to one round trip — or zero with resumption. Better for lossy networks like mobile.
>
> Server push existed in HTTP/2 but is effectively dead — Chrome removed support — because it was hard to use without wasting bandwidth. The replacement is `103 Early Hints` plus `preload`."

---

### Follow-up 5 — "What's the difference between `DOMContentLoaded`, `load`, and LCP?"

> "`DOMContentLoaded` fires when the HTML has been parsed and the DOM tree is built — synchronous scripts have run, but stylesheets, images, and subframes may still be loading.
>
> `load` fires when _everything_ — images, stylesheets, iframes — has finished. It's a much later milestone and not usually what users experience as 'loaded.'
>
> LCP — Largest Contentful Paint — is a Core Web Vital. It measures when the largest visible content element in the viewport paints. This usually correlates with what a user perceives as 'the page is here.' Targets: under 2.5s is good.
>
> They answer different questions: `DOMContentLoaded` for 'can my JS start running,' `load` for 'is everything fetched,' LCP for 'does the user see the page.'"

---

### Follow-up 6 — "Walk me through `async` vs `defer` vs neither on a script tag."

> "All three describe how a `<script>` tag interacts with the HTML parser:
>
> - **No attribute** — the parser stops, downloads the script, executes it, then resumes parsing. Worst for performance, blocks first paint.
> - **`async`** — script downloads in parallel with parsing, then executes as soon as it's ready, pausing the parser at that moment. Order of execution is not guaranteed across multiple async scripts. Good for independent third-party scripts like analytics.
> - **`defer`** — script downloads in parallel with parsing, but execution is deferred until _after_ parsing finishes, in document order. This is what you usually want for first-party scripts that need the DOM.
>
> Modules (`type="module"`) are deferred by default."

---

### Follow-up 7 — "How does the browser decide when to use HTTP/3?"

> "Hard problem because HTTP/3 is over UDP and the browser can't know upfront if the server supports it. The mechanism is `Alt-Svc` — the browser connects over HTTP/2 first, the server responds with an `Alt-Svc` header advertising HTTP/3 endpoints, and the browser tries QUIC for subsequent requests. If QUIC is blocked by a firewall, it falls back to HTTP/2.
>
> Some browsers and CDNs are experimenting with DNS-based discovery via HTTPS resource records (RFC 9460) so the very first connection can go to QUIC."

---

### Follow-up 8 — "What happens if DNS fails?"

> "Several failure modes:
>
> - **Recursive resolver times out** — browser shows `DNS_PROBE_FINISHED_NO_INTERNET` or similar. Browsers often have fallback resolvers configured.
> - **NXDOMAIN** — domain doesn't exist. ISPs sometimes hijack this for ad pages, which is part of why DoH is getting popular.
> - **Stale cached record** — pointing at an IP that no longer serves the site. Mitigated by short TTLs but those have their own cost.
>
> At the application layer, browsers can preconnect or prefetch DNS for likely-next destinations using `<link rel="dns-prefetch">` or `<link rel="preconnect">`."

## Part 3: Things to Drop Naturally for Senior Signal

Sprinkle these in where they fit. They demonstrate currency:

- **HTTP/3 / QUIC** — over UDP, no TCP head-of-line blocking
- **TLS 1.3** — 1-RTT, 0-RTT with resumption
- **Encrypted Client Hello (ECH)** — replacing ESNI, hides hostname from network observers
- **Brotli / zstd** — modern compression beating gzip
- **Core Web Vitals: LCP, INP, CLS** — INP replaced FID in 2024
- **CDN edge compute** — Cloudflare Workers, Lambda@Edge — shifting logic out of the origin
- **`103 Early Hints`** — replaces HTTP/2 server push
- **AVIF / WebP** — modern image formats
- **DoH / DoT** — encrypted DNS
- **Certificate Transparency** — public logs of all issued certs

## Part 4: Quick-Reference Mental Model

If you blank, anchor on this seven-word skeleton:

> **Parse → Cache → DNS → TCP → TLS → HTTP → Render**

For each one, you should be able to answer: _what does it do, what can fail, and what makes it faster on the second visit?_

If you can answer that for all seven, you have a complete senior-level answer to this question.
